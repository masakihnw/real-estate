# 7人の意地悪なQA（テスト観点漏れ撲滅スキル）

> 出典の発想: 「Claude Code に『7人の意地悪なQA』を仕込んでテストケースの観点漏れを潰した」(Ayaka)。
> 本スキルはその型を **この real-estate プロジェクト**（iOS SwiftUI / Python スクレイピング・enricher /
> Supabase 移行 / AI購入分析）の文脈に翻案したもの。

## いつ使うか

- 変更箇所のユニットテスト / レグレッションテストを設計するとき（CLAUDE.md「実装中はこまめにユニットテスト」）。
- `code-reviewer` agent によるレビュー（CLAUDE.md「方針確定時と実装完了時の2回必須」）の観点チェックリストとして。
- パーサ / enricher / マイグレーション / iOS ロジックの PR を出す前のセルフレビューとして。

## 中核スタンス

**「動いているはず」を信用しない。意地悪なくらい疑ってかかる。**
各ペルソナが **最低1つ** は確認観点を残す。観点が思いつかない＝まだ疑い足りない、と捉える。

## 7人のQAペルソナ（この repo 用に翻案）

| # | ペルソナ | このプロジェクトで疑う点（代表観点） | 主な対象 |
|---|---------|-----------------------------------|---------|
| **P1** | 新人ユーザー | 説明を読まず直感操作。誤タップ・空入力・連打・オフライン/電波断・初回起動（空データ）で iOS 画面が壊れないか。和暦端末・ロケール差（`DateFormatter` の `en_US_POSIX` 共有）で表示が崩れないか | iOS UI/状態 |
| **P2** | ベテラン現場担当 | 大量データ・高速操作。ウォッチリスト/比較が大量件数でも破綻しないか。フィルタ多用・連続スクロール・バックグラウンド復帰後の整合（`ios-performance-investigation` 参照） | iOS 性能/状態 |
| **P3** | 悪意ある/異常入力者 | 境界値・不正値・想定外フォーマット。価格0/欠損/全角数字/「-」/単位ゆれ、壊れた・改変された HTML、文字コード混在。bot ブロックの偽装ページ。パース例外を握り潰していないか | スクレイパー/パーサ |
| **P4** | データ整合性監査役 | 画面・ログを信用せず**裏のDBを直接確認**。Supabase/Firestore の CRUD 整合、dedup の取りこぼし/誤統合、delisting/grace period の整合、`upsert` の冪等性、version/prompt_hash の正しさ | DB/dedup/enrich |
| **P5** | 移行担当者 | 既存データ投入。Firebase→Supabase 移行・マイグレーションでの欠損・異形式・件数一致。マイグレーション採番（既存最大+1、3桁、過去025衝突）。**新機能の作り方で移行を設計しない**（要件は満たすが既存仕様とズレるケースを取りこぼす） | 移行/migrations |
| **P6** | 回帰デグレ番人 | 「今まで動いていた何か」が壊れていないか。`tests/test_pipeline_smoke.py` のステージ間結線、単一ソース同期（後述）、既存ユニットテスト、iOS 既存ロジック。**これが CLAUDE.md「デグレ防止」の核** | 全体/回帰 |
| **P7** | 仕様懐疑者 | 「実装＝正しい仕様」を信用しない。**単一ソース**と実挙動を突合: `ScrapingConfigMetadata.json` / `docs/SPECIFICATION.md`、`buyer_profile.json` / `purchase_strategy.md` / `prompts/<module>.md` / `docs/BUYER_PROFILE.md`、Supabase 本番値。フェイルクローズ原則（取りこぼし側に倒す）が守られているか | 仕様/単一ソース |

### そのままプロンプトに貼れる版

```
あなたは「動いているはず」を信用しない7人のQAです。
各ペルソナの視点で、この変更を最低1つずつ壊しにいってください。
- P1 新人：誤タップ・空入力・連打・オフライン・初回起動・和暦/ロケール
- P2 ベテラン：大量データ・高速操作・バックグラウンド復帰後の整合
- P3 異常入力：価格0/欠損/全角数字・壊れたHTML・文字コード・bot偽装ページ・握り潰し例外
- P4 データ整合：画面でなく裏のDB(Supabase/Firestore)を直接確認。dedup・delisting・冪等upsert
- P5 移行：旧データ投入。欠損・件数一致・マイグレ採番。新機能の作り方で移行を設計しない
- P6 回帰：pipeline smoke・単一ソース同期・既存テスト・既存iOSロジックが壊れていないか
- P7 仕様懐疑：単一ソース(設定/戦略/仕様書/本番)と実挙動を突合。フェイルクローズが効くか
```

## 仕様の出どころで観点の軸を分ける（新機能 / 移行・構造変更）

同じ「テストする」でも、仕様の出どころが違えば正解の基準も変わる。混ぜると出力がちぐはぐになる。

| 観点 | 新機能・通常変更 | 移行・スクレイピング構造変更 |
|------|----------------|---------------------------|
| 仕様の出どころ | 課題/PR の要件 ＋ 該当ソースコード | 既存挙動・既存データ ＋ 単一ソース（設定/戦略/仕様書） |
| 正解の基準 | 要件を満たしていること | **既存仕様どおり**に動き、**取りこぼしがない**こと |
| 主役ペルソナ | P3 異常入力 ＋ P4 データ整合 | P5 移行 ＋ P6 回帰 ＋ P7 仕様懐疑 |
| 必須テスト | 純関数ユニットテスト | 同上 ＋ `test_pipeline_smoke.py` ＋ 件数/差分検証 |

## 根拠のないケース・観点を出さない（偽陽性を出さない）

テスト設計と同じで、**それらしいけれど根拠のない**ケースを量産しないことが最優先。

- **一次情報を引く**: 新機能なら課題/PR 番号・該当ソースの関数名、移行/設定なら単一ソースの該当キー
  （`ScrapingConfigMetadata.json` のフィールド名等）を観点の根拠として明示する。独自採番・推測の仕様を作らない。
- **確認できていないことは「未確認」と正直に書く**。コード・DBで裏取りできていない観点は
  「※要確認（未実施）」と残し、断定しない。**「分からないことは分からないと書く」が一番効く。**
- **やらないことを明示する**: 「このレビューは指摘のみ・修正はしない」「裏取りできない箇所は埋めない」
  といった制約を宣言してから進める（AIの越権・でっち上げを抑える）。

## 出力の型（テスト観点を残すとき）

テスト設計の成果物（テスト観点表・PR説明のテスト節）には、最低限こう残す:

- **どのペルソナ視点か**（P1〜P7）と**疑った点**。
- **根拠（一次情報）**: 課題/PR番号 or 単一ソースの該当キー or ソース関数名。
- **期待結果**: 特に P4 は「裏のテーブル/カラムに正しい値が入るか」まで書く。
- **回帰対象か（YES/NO）**: P6 観点。触ると壊れうる周辺（pipeline smoke・単一ソース同期）を明記。
- **品質特性（ISO/IEC 25010, 2023年版 9特性）**で観点の偏りを可視化:
  機能適切性 / 性能効率性 / 互換性 / 使用性 / 信頼性 / セキュリティ / 保守性 / 移植性 / **安全性**。

## このスキルと CLAUDE.md の対応

- 「デグレ防止」「レグレッションテスト必須」→ **P6 回帰デグレ番人** を毎回必ず通す。
- 「設定の単一ソース」「買い手コンテキストの正準ソース」→ **P7 仕様懐疑者** で突合。
- 「フェイルクローズ（取りこぼし側に倒す）」「パース0件の終端/ブロック区別」→ **P3 / P7**。
- 「`code-reviewer` agent を2回必須」→ 本ペルソナ表をそのレビューのチェックリストに使う。
