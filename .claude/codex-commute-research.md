# Codex 調査依頼: 通勤時間取得の代替手段

## 背景

不動産物件検索アプリで、各物件の最寄り駅から2つのオフィスへの通勤時間を自動取得したい。

### 現状
- Yahoo路線情報の Web ページを WebFetch でスクレイピングしていた
- 実行環境が「Claude Desktop Routines」（Anthropic のリモートサーバー上で動作）に移行
- リモート環境から Yahoo Transit へのアクセスが **HTTP 403 Forbidden** でブロックされる
- 利用可能なツール: **WebFetch**（URL → HTML/JSON 取得）、**WebSearch**（Web検索）、**Supabase MCP**（DB操作）
- **Playwright / ブラウザ操作は不可**（リモート環境制約）

### 要件
- **入力**: 最寄り駅名（例: 「石神井公園」「西大島」「勝どき」）
- **出力**: 2つの目的地への通勤時間・乗り換え回数・ルート概要
  - **playground**: 最寄駅 半蔵門駅（東京メトロ半蔵門線）。実住所は環境変数で管理
  - **m3career**: 最寄駅 神谷町駅（東京メトロ日比谷線）。実住所は環境変数で管理
- 朝の通勤時間帯（9:00 到着想定）の実所要時間が理想
- 対象は東京23区 + 周辺（神奈川・埼玉・千葉の一部）の鉄道駅
- 月あたり ~200 駅程度のクエリ（新規物件追加分）

### 精度要件
- 乗り換え時間込みの「ドアtoドア」所要時間が理想だが、「駅to駅」の所要時間でも十分
- 乗り換え回数が分かると望ましい
- ±5分程度の誤差は許容

## 調査してほしいこと

### 1. 日本の乗り換え案内 API（有料含む）
- **Google Maps Directions API** (Transit mode): 料金体系、日本の鉄道対応状況、精度、レート制限
- **NAVITIME API**: 法人向け API の有無、料金、WebFetch でアクセス可能か
- **駅すぱあと Web サービス API** (Val Laboratory): 料金、個人開発者向けプラン
- **ジョルダン API**: 同上
- **その他**: RESAS API、国土交通省オープンデータ等で使えるものがあるか

### 2. 無料 or 低コストの代替案
- **Google Maps URL → WebFetch**: `https://www.google.com/maps/dir/...` のようなURLでルート情報が取得できるか（403 にならないか）
- **Apple Maps / MapKit**: API でルート検索が可能か
- **GTFS（General Transit Feed Specification）データ**: 日本の鉄道の GTFS データは公開されているか。ローカルでルート計算するアプローチの実現可能性
- **OpenTripPlanner + 日本 GTFS**: セルフホストの乗り換え案内。Supabase Edge Function で動かせるか

### 3. 静的アプローチ（APIなし）
- **駅間所要時間マスターテーブル**: 主要駅（半蔵門・神谷町）への所要時間を事前計算して Supabase テーブルに持つ。更新頻度はどうすべきか
- **路線ネットワークグラフ + ダイクストラ**: 駅間の接続と所要時間をグラフ化して最短経路を計算。データソースは何を使えるか
- **既存の駅間所要時間データセット**: Kaggle、GitHub 等で日本の鉄道駅間所要時間のデータセットはあるか

### 4. 実装上の制約との適合性
各手段について以下を評価:
- Claude Desktop Routines の **WebFetch** でアクセス可能か（403 にならないか）
- API キーが必要な場合、WebFetch のヘッダーに付けられるか or Supabase Edge Function 経由で呼べるか
- 月額コストの目安（~200 クエリ/月の想定）
- セットアップの手間
